# 03 JVM与性能调优（零基础版）

## 先用人话说清楚
- JVM 就像汽车发动机，GC 就像自动打扫卫生。
- 服务变卡，通常是发动机忙不过来，或垃圾太多清不动。

## 术语翻译
- 堆：放对象的大仓库。
- 栈：每个线程自己的小工作台。
- GC：回收没用对象的机制。
- STW：打扫时让应用短暂停一下。
- P99：100次请求里最慢的那几次，最能反映卡顿。

## 1. JVM 内存和 GC
### 是什么
- JVM 管理对象存储和回收，避免手动释放内存。

### 为什么
- 内存用不好会导致频繁 GC、延迟抖动、甚至服务假死。

### 怎么做
- 先监控：GC 次数、停顿时间、堆使用率。
- 再优化：减少大对象、减少短时间暴增对象。

### 高并发如何选择
- 低延迟业务优先减少停顿；吞吐业务优先整体处理量。

### 面试题与标准答案
- 问：为什么会频繁 Full GC？
- 答：常见是老年代压力大、对象晋升快或有泄漏，要先看 GC 日志和堆快照定位。

## 2. 性能分析方法
### 是什么
- 用数据定位慢点，而不是靠感觉调优。

### 为什么
- 错误调优会把系统越改越慢。

### 怎么做
- 固定流程：先基线 -> 压测 -> 定位热点 -> 单变量优化 -> 回归验证。

### 高并发如何选择
- 先解决 P99 长尾，再看平均值。

### 面试题与标准答案
- 问：性能优化第一步是什么？
- 答：先量化现状并建立基线，明确瓶颈在 CPU、内存、IO 还是锁。

## 3. 线上排障
### 是什么
- 在故障发生时快速止损并定位根因。

### 为什么
- 高级岗位必须能在压力下做正确动作。

### 怎么做
- 先止损：限流、降级、回滚。
- 再定位：线程栈、堆快照、火焰图。

### 高并发如何选择
- 先保核心接口可用，非核心功能可临时降级。

### 面试题与标准答案
- 问：CPU 飙高你先做什么？
- 答：先确认是否影响核心链路并限流止损，再抓线程和火焰图定位热点代码。

## 本章30秒复述
- JVM 问题本质是内存和执行效率问题。
- 调优必须数据驱动。
- 故障处理顺序是先止血再根因。

## 追根究底：JVM 背后的物理机与操作系统

### 1) 为什么对象分配快，但 Full GC 很痛
### 是什么
- 大多数对象在堆上按指针碰撞快速分配，速度接近顺序写内存。

### 为什么
- 年轻代回收快是因为“多数对象很快死亡”；但老年代回收需要遍历和整理，成本高。

### 怎么做
- 控制对象生命周期，减少短时大量临时对象。
- 减少大对象直接进入老年代的概率。

### 高并发如何选择
- 低延迟业务优先降低分配抖动和晋升压力，而不是盲目增大堆。

### 面试题与标准答案
- 问：为什么堆调大了，有时反而更卡？
- 答：堆变大降低了回收频率，但单次回收扫描范围变大，停顿可能更长。

### 2) CPU 缓存与指令流水线如何影响 Java 性能
### 是什么
- CPU 真正执行时依赖缓存和流水线，不是逐条慢慢执行字节码。

### 为什么
- 分支预测失败、缓存未命中、内存随机访问都会让 CPU 空转等待。

### 怎么做
- 热路径数据结构尽量连续访问，减少随机指针跳转。
- 避免在核心循环里做复杂对象装箱拆箱。

### 高并发如何选择
- 对延迟敏感场景，优先优化缓存命中率和对象布局。

### 面试题与标准答案
- 问：为什么同样 O(n) 的两段代码，耗时差很多？
- 答：大 O 只描述增长趋势，真实性能还受缓存命中、分支预测、常数项影响。

### 3) 线程阻塞为什么会放大延迟
### 是什么
- 线程被阻塞时会触发内核调度，发生上下文切换。

### 为什么
- 切换有系统开销，阻塞多了会造成线程池排队和级联超时。

### 怎么做
- 区分 CPU 密集和 IO 密集线程池。
- 为外部依赖设置超时，避免线程长期占用。

### 高并发如何选择
- 热链路以“少阻塞、少等待”为先，必要时异步化。

### 面试题与标准答案
- 问：线上线程池满了最先看什么？
- 答：先看任务类型和阻塞点，是慢 SQL、慢 RPC 还是锁等待，再决定扩容、限流或降级。
