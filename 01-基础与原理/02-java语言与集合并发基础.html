02 Java语言与集合并发基础

先用人话把这章说清楚

  • 你可以把“并发”理解成：一个餐厅同时来很多客人，后厨怎么不乱。
  • 程序里的“线程”就是多个小厨师；“锁”就是同一口锅同一时间只能一个人用。
  • 这章核心不是背术语，而是学会回答两个问题：
    • 为什么会乱？
    • 怎么让它快而且不出错？

术语翻译（先看这个）

  • 线程：程序里同时干活的“工人”。
  • 共享变量：大家都能改的一张“公共白板”。
  • 可见性：A 改了白板，B 能不能马上看到。
  • 原子性：一个动作要么全做完，要么不做，不能只做一半。
  • 锁：同一时间只允许一个人改白板。
  • 线程池：提前招好一批工人，来活就派，不临时招人。
  • 队列：排队等处理的任务清单。

1. Java内存模型（JMM）与可见性

是什么

  • 可以把 JMM 理解成“多人协作写白板的规则书”。
  • 它规定了：谁改了数据，别人什么时候能看到，以及动作顺序会不会被打乱。

为什么

  • 不懂这个规则，就会出现“你电脑上没问题，一上线上多人同时访问就出错”。
  • 这种问题最难查，因为偶发、难复现。

怎么做

  • 第一步：先记住一个原则，多个线程改同一份数据，必须有同步手段。
  • 第二步：简单状态用 volatile；复杂修改（如 i++）用锁或原子类。
  • 第三步：写并发代码后做压力测试，不要只跑单线程。

高并发如何选择

  • 读很多、写很少、逻辑简单：先选 volatile。
  • 会涉及多步修改、必须绝对正确：先选锁。
  • 冲突不高、追求性能：再考虑 CAS。

面试题与标准答案

  • 问：为什么 volatile 不能保证 i++ 正确？
  • 答：因为 i++ 是“读-改-写”三步，不是原子操作。volatile 只保证看得见，不保证三步合成一步。
  • 问：什么时候必须上锁？
  • 答：只要对共享数据有复合操作，并且结果必须准确，就要上锁或用原子类保证原子性。

2. 线程池与异步编排

是什么

  • 线程池就是“固定工人团队”。任务来了直接分配，不用每次临时创建线程。

为什么

  • 线程创建很贵；无限创建会把机器拖垮，响应变慢甚至崩掉。

怎么做

  • 按任务类型分池：计算任务一组，IO任务一组。
  • 队列必须有上限，不能无限堆。
  • 必须有拒绝策略：忙不过来时要快速失败或降级。

高并发如何选择

  • 要求快响应：小队列，满了就拒绝，保护系统。
  • 要求多处理：队列可以稍大，但要配合超时和降级。

面试题与标准答案

  • 问：为什么不建议用无界队列？
  • 答：因为任务会无限堆积，内存被吃光，最终触发雪崩。
  • 问：线程池最少要看哪些监控？
  • 答：活跃线程数、队列长度、任务拒绝数、任务执行耗时。

3. 集合与并发容器

是什么

  • 容器就是“装数据的盒子”，并发容器是“多人同时用也不容易出错的盒子”。

为什么

  • 容器用错了，会出现数据错乱、性能暴跌、CPU 飙高。

怎么做

  • 多线程场景优先用并发容器，不要直接用普通 HashMap。
  • 读多写少才用 CopyOnWriteArrayList；写多会非常慢。

高并发如何选择

  • 高频读写混合：ConcurrentHashMap。
  • 大量排队消费：BlockingQueue。

面试题与标准答案

  • 问：为什么多线程下不能直接用 HashMap？
  • 答：因为它不是线程安全的，可能出现数据覆盖、丢失，甚至死循环风险（历史实现中）。
  • 问：CopyOnWriteArrayList 适合什么场景？
  • 答：读很多写很少，比如配置快照、白名单读取。

4. 锁优化与无锁思路

是什么

  • 锁优化就是“尽量减少抢同一把锁”，无锁思路是“尽量让大家不互相堵”。

为什么

  • 锁一多，线程就排队，系统变慢，延迟升高。

怎么做

  • 缩小上锁范围：只锁真正需要保护的几行代码。
  • 数据分片：把一把大锁拆成多把小锁。
  • 热点请求先削峰，再慢慢处理。

高并发如何选择

  • 冲突不高：CAS 能更快。
  • 冲突很高：用分片锁或队列串行，避免 CAS 一直自旋浪费 CPU。

面试题与标准答案

  • 问：CAS 一定比锁快吗？
  • 答：不一定。冲突低时通常快；冲突高时会大量重试，反而更慢。
  • 问：锁优化第一步做什么？
  • 答：先定位热点锁（数据驱动），再做缩小临界区和分片改造。

本章5分钟复述模板（可直接背）

  • 并发问题本质是多人同时改同一份数据。
  • 先保证正确，再追求性能：
    • 简单可见性用 volatile。
    • 复合操作用锁或原子类。
    • 任务执行用线程池，不要无限建线程。
    • 容器要按读写比例选。
    • 高并发下通过分片、削峰、限流减少冲突。

追根究底：物理机 / 操作系统 / 数据结构

1) 物理机视角：为什么 volatile 和锁会“生效”

是什么

  • 多核 CPU 下，每个核心都有缓存；线程看到的值不一定立刻一致。

为什么

  • 如果没有缓存一致性协议和内存屏障，线程 A 写入后线程 B 可能看不到最新值。

怎么做

  • volatile：通过内存屏障约束重排并刷新可见性。
  • 锁：进入/退出临界区时形成更强的可见性和有序性保证。

高并发如何选择

  • 单变量状态同步：volatile。
  • 多变量一致性与复合写：锁或原子类。

面试题与标准答案

  • 问：为什么说 volatile 只能保可见性不能保原子性？
  • 答：因为它解决的是“看得见”和“顺序”，不把读-改-写三步合成不可分割的一步。

2) 操作系统视角：为什么线程多了反而更慢

是什么

  • 线程切换需要保存/恢复执行现场，属于真实系统开销。

为什么

  • 线程过多会导致上下文切换、锁竞争、缓存失效同时上升。

怎么做

  • 线程池按任务类型分离：CPU 密集与 IO 密集分开。
  • 用有界队列控制积压，避免把延迟问题变成内存问题。

高并发如何选择

  • 延迟敏感：小队列 + 快失败。
  • 吞吐敏感：适度队列 + 降级兜底。

面试题与标准答案

  • 问：为什么线程不是越多越好？
  • 答：超过系统承载后，切换和争抢成本会吃掉收益，吞吐下降、延迟升高。

3) 数据结构视角：为什么 ConcurrentHashMap 能抗并发

是什么

  • 它用哈希桶分散冲突，并把写锁范围尽量缩小到局部。

为什么

  • 全局一把大锁会让所有线程排队，局部同步能提升并行度。

怎么做

  • key 设计要均匀，减少热点桶。
  • 高频更新场景用分片键降低冲突。

高并发如何选择

  • 高读高写：ConcurrentHashMap。
  • 高频排队消费：BlockingQueue。

面试题与标准答案

  • 问：ConcurrentHashMap 比 HashMap 快在哪里？
  • 答：不是单纯“快”，而是在并发场景下通过局部同步和更合理的结构减少冲突与数据错误。

4) 一条可复述的根因链

  • 现象：接口偶发超时、CPU 升高。
  • 根因：线程暴涨 -> 切换增多 -> 锁竞争上升 -> 队列堆积 -> 延迟抖动。
  • 处理：限流 + 线程池收敛 + 热点分片 + 缩临界区。
