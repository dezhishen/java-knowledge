# Kafka（Java Web 进阶）

面向服务端开发：把 Kafka 当作「分布式提交日志（Distributed Log）+ 高吞吐消息系统」来理解。它的核心不是“消息队列”四个字，而是 **顺序追加、分区并行、批量传输、可重放消费** 这套工程能力。

---

## 1. Kafka 是什么：先看能力边界

- **核心定位**：高吞吐、可持久化、可回放的事件流平台。
- **适合场景**：异步解耦、削峰填谷、日志采集、行为埋点、流式处理、事件驱动架构。
- **不擅长**：低延迟强事务请求-响应、复杂路由（比 RabbitMQ 弱）、超长延迟任务调度（要配合其他组件）。

一句话：Kafka 更像“可订阅的日志”，不是“只能读一次的传统队列”。

---

## 2. 核心模型：Topic、Partition、Offset

### 2.1 Topic 与 Partition

- **Topic**：逻辑主题，表示一类事件（如 `order-created`）。
- **Partition**：Topic 的物理分片，每个分片内部有序、分片之间无全局顺序。
- **并行能力来源**：吞吐主要靠分区扩展，不是靠单分区加速。

### 2.2 Offset

- 每条消息在分区内有递增的 `offset`。
- 消费者本质是“读取某分区某 offset 之后的数据”。
- 因为 offset 可控，Kafka 天生支持“重放历史数据”。

---

## 3. 为什么 Kafka 吞吐高

- **顺序写磁盘**：追加写对磁盘/SSD 更友好，随机写少。
- **批量发送与拉取**：Producer/Consumer 都是批量 I/O，减少系统调用与网络往返。
- **零拷贝（sendfile）**：减少用户态/内核态数据复制开销。
- **分区并行**：多分区可跨 Broker、跨 CPU 核并行处理。
- **页缓存利用**：热数据常驻 OS page cache，读写命中高时延迟更稳。

工程直觉：Kafka 的“快”是磁盘顺序 I/O + 批处理 + 网络优化共同结果，不是“全内存”方案。

---

## 4. 复制与高可用：Leader/Follower 与 ISR

- 每个分区有 1 个 **Leader**，多个 **Follower**。
- 生产与消费都只面向 Leader，Follower 负责复制数据。
- **ISR（In-Sync Replicas）**：与 Leader 保持同步的一组副本。
- Leader 挂掉后，从 ISR 中选新 Leader，保证可用性。

常见参数理解：

- `replication.factor`：副本数（生产常见 3）。
- `min.insync.replicas`：最小同步副本数，配合 `acks=all` 决定写入可靠性下限。

---

## 5. 生产者可靠性语义

### 5.1 acks 语义

- `acks=0`：不等确认，吞吐高，最不可靠。
- `acks=1`：Leader 写入即返回，Leader 刚写完就宕机可能丢数据。
- `acks=all`：ISR 全部确认才返回，可靠性最高（结合 `min.insync.replicas`）。

### 5.2 重试与幂等

- 网络抖动下会重试，可能导致重复写。
- 开启 `enable.idempotence=true` 后，单生产者会话内可避免重复写入同分区。
- 业务端仍要做幂等（比如按业务主键去重），不要把“绝不重复”只寄托在 MQ。

### 5.3 事务消息（Exactly Once 的一部分）

- Kafka 事务可把“写多个分区 + 提交消费位点”绑定成原子提交。
- 适合 Kafka Streams / 消费-处理-再生产链路。
- 代价是复杂度和延迟上升，普通业务先保证“至少一次 + 业务幂等”通常更划算。

---

## 6. 消费者与消费组机制

### 6.1 Consumer Group

- 同一 group 内：一个分区同一时刻只会分配给一个消费者实例。
- 不同 group 间：可各自独立消费同一份数据（天然多订阅者）。

### 6.2 Rebalance（重平衡）

- 触发时机：实例增减、订阅变化、心跳超时。
- 影响：分区重新分配，短时间暂停消费，可能导致延迟抖动。
- 实战建议：控制实例频繁上下线，优先稳定部署，减少无效 rebalance。

### 6.3 位点提交（Offset Commit）

- **自动提交**：简单，但故障时重复/丢失窗口更难精确控制。
- **手动提交**：处理成功后再提交，语义更可控（常用于重要业务）。

---

## 7. 消息投递语义与业务落地

- **At most once**：先提交位点再处理，可能丢消息。
- **At least once**：先处理再提交位点，可能重复消息（最常用）。
- **Exactly once**：依赖 Kafka 事务 + 端到端配合，成本高。

Java 业务里最常见策略：

1. 采用至少一次投递；
2. 消费端实现幂等（唯一键、去重表、状态机防重）；
3. 失败重试 + 死信队列，保证“最终处理完成”。

---

## 8. 顺序性、积压与扩缩容

### 8.1 顺序性

- Kafka 只保证 **分区内顺序**，不保证 Topic 全局顺序。
- 需要同一业务键顺序时，用相同 key（如 `orderId`）路由到同一分区。

### 8.2 消息积压

- 现象：`consumer lag` 持续增长。
- 常见原因：消费慢、下游依赖慢、分区数不足、单条消息处理过重。
- 处理思路：扩消费者实例（不超过分区数）、增加分区、拆分主题、优化消费逻辑与批处理。

### 8.3 分区扩容注意

- 增加分区会改变 key 到分区的映射（对顺序/热点有影响）。
- 对“强依赖分区顺序”的业务，扩分区前要评估迁移窗口和兼容策略。

---

## 9. 关键参数（入门实战版）

### 9.1 Producer

- `acks=all`
- `enable.idempotence=true`
- `retries`（允许重试）
- `batch.size`、`linger.ms`（批量与延迟折中）
- `compression.type`（常用 `snappy`/`lz4`/`zstd`）

### 9.2 Consumer

- `enable.auto.commit=false`（重要链路建议手动提交）
- `max.poll.records`（单次拉取条数）
- `max.poll.interval.ms`（处理超时相关）
- `session.timeout.ms`、`heartbeat.interval.ms`（心跳与存活判定）

### 9.3 Broker/Topic

- `num.partitions`
- `replication.factor`
- `min.insync.replicas`
- `retention.ms` / `retention.bytes`（保留策略）

## 10. Kafka 与 Redis/RabbitMQ 的简单对比

- **Kafka vs Redis Stream/List**：Kafka 更适合大吞吐、可回放、长期保留；Redis 更轻量、低延迟、适合中小规模。
- **Kafka vs RabbitMQ**：Kafka 强在吞吐、日志流、重放；RabbitMQ 强在复杂路由、协议支持和低延迟任务分发。
- **选型原则**：看业务目标优先级——吞吐与可回放优先选 Kafka；复杂路由和即时任务分发可优先 RabbitMQ。

---

## 11. Java Web 落地 checklist

- **事件建模**：先定义事件名、版本、关键字段，避免后期 schema 混乱。
- **消息体规范**：使用 JSON/Avro/Protobuf 并管理版本兼容（新增字段向后兼容）。
- **生产端**：发送失败要记录可追踪日志，必要时本地落盘或重试队列兜底。
- **消费端**：业务处理与位点提交顺序明确；异常分可重试和不可重试两类。
- **幂等设计**：按业务唯一键去重（如订单号 + 事件类型）。
- **监控告警**：重点看 TPS、失败率、Consumer Lag、Rebalance 次数、请求延迟 P99。
- **容量规划**：按峰值流量估算分区数和副本数，预留扩容空间。

---

## 12. 常见面试/排障问法速答

- **为什么会重复消费？**  
  因为至少一次语义下，处理成功但位点未提交（或提交失败）后重启会重读。

- **如何保证消息不丢？**  
  生产端 `acks=all` + 合理副本与 ISR；消费端处理成功再提交；关键路径加重试与补偿。

- **如何保证顺序？**  
  同一业务 key 固定分区，分区内单线程（或有序）处理。

- **Lag 很高怎么办？**  
  先查消费端瓶颈（CPU/IO/下游），再考虑扩实例、增分区、批处理优化。

---

## 13. 一句话总结

Kafka 的本质是“可重放的分布式日志”。在 Java Web 里，最实用的落地套路不是追求理论上的绝对一次，而是 **至少一次 + 幂等 + 可观测 + 补偿机制**，用工程手段拿到稳定性与吞吐的平衡。
