14 面试问答标准答案（零基础版）

使用方法

  • 每题先自己说 30 秒，再对照标准答案。
  • 如果答不上来，回看对应章节的“术语翻译”和“怎么做”。

A. 并发与Java基础

问：volatile 和 synchronized 的区别是什么？

  • 标准答：volatile 保证可见性和有序性，但不保证复合操作原子性；synchronized 同时保证可见性和原子性，适合对共享资源做完整保护。

问：线程池为什么不能随便开很大？

  • 标准答：线程越多并不一定越快，超过 CPU/IO 承载后会引发频繁切换和资源争抢，导致延迟上升。

B. JVM

问：为什么会发生 Full GC，怎么处理？

  • 标准答：常见原因是老年代压力过大、对象晋升过快或内存泄漏。处理顺序是先看 GC 日志和堆快照定位热点，再做对象生命周期优化、参数调优或代码修复。

C. Spring

问：事务为什么会失效？

  • 标准答：最常见是同类内部调用绕过代理、方法不是 public、异常被吃掉、传播行为配置不当。

D. MySQL

问：为什么加了索引还慢？

  • 标准答：可能没命中索引、回表太多、排序分组无法利用索引、返回行太大。要先看 EXPLAIN 再调整 SQL 和索引。

E. Redis

问：怎么防止缓存雪崩？

  • 标准答：过期时间打散、热点数据预热、多级缓存、限流降级、故障时分批恢复，避免流量瞬间打到数据库。

F. Kafka

问：消息重复消费怎么办？

  • 标准答：把幂等放在业务层，使用业务唯一键、去重表或状态机，保证重复处理不会造成错误结果。

G. 微服务治理

问：什么时候做熔断，什么时候做降级？

  • 标准答：下游异常率高或超时上升时先熔断保护主系统；资源不足时对非核心功能做降级，保障主交易路径。

H. 高并发设计

问：高并发系统第一原则是什么？

  • 标准答：先保正确和稳定，再保性能。先解决雪崩风险，再优化吞吐和延迟。

I. 系统设计

问：面试系统设计先讲什么？

  • 标准答：先澄清需求和指标（QPS、延迟、可用性），再给架构、数据模型、一致性方案、容灾和监控。

J. 根因追问（物理机/OS/数据结构）

问：为什么线程不是越多越快？

  • 标准答：线程过多会增加上下文切换和锁竞争，CPU 花在调度上的时间变多，吞吐反而下降。

问：为什么 B+Tree 适合数据库索引？

  • 标准答：B+Tree 节点能容纳更多键，树高低，磁盘访问次数少，且天然支持范围查询。

问：Kafka 吞吐高的底层原因是什么？

  • 标准答：核心是顺序追加写、页缓存协同、批量传输和减少拷贝/系统调用。

问：为什么 Spring 里同类调用会让事务失效？

  • 标准答：因为事务拦截依赖代理对象，同类内部通过 this 调用不会经过代理，拦截器链不会触发。

问：为什么 Redis 有时会出现“明明有缓存却还是慢”？

  • 标准答：热点 key、淘汰抖动、网络与序列化开销都可能让瓶颈不在 Redis 命令本身，而在回源和传输成本。

问：为什么分布式系统不能只靠时间戳判断顺序？

  • 标准答：机器时钟存在偏差，网络延迟不稳定，时间戳只可参考，关键顺序应靠版本号/序列号/幂等键保证。
