    Redis（Java Web 进阶）¶

    面向服务端开发：把 Redis 当作「高性能内存数据结构服务 + 可选持久化 + 分布式协调组件」来理解。它快，不只因为在内存，还因为事件循环、I/O 多路复用、数据结构设计与场景边界控制。

    1. Redis 是什么：能力边界先划清¶

    • 核心定位：内存 KV 存储，支持丰富数据结构（String、Hash、List、Set、ZSet、Stream、Bitmap、HyperLogLog、Geo）。
    • 典型用途：缓存、计数器、排行榜、分布式锁、会话存储、限流、消息队列（轻量）。
    • 不擅长：复杂多表关系查询、强事务一致性场景、超大数据集离线分析。

    一句话：Redis 适合「高频、短小、可容忍一定最终一致」的读写路径，不是拿来替代 MySQL 的万能数据库。

    2. 为什么快：从 CPU 与网络 I/O 看¶

    • 内存访问：避免磁盘随机 I/O，延迟主要来自网络 RTT + 内存访问 + 命令执行。
    • 单线程命令执行模型（核心路径）：减少锁竞争和线程切换，避免多线程共享状态的复杂同步。
    • I/O 多路复用：基于 epoll/kqueue 同时管理大量连接，事件就绪后快速处理。
    • 高效数据结构：多数命令时间复杂度低（如 GET 平均 O(1)、HGET 平均 O(1)、ZADD O(logN)）。
    • 避免阻塞：Redis 主线程尽量做短任务；慢命令和大 key 会直接放大尾延迟。

    工程直觉：Redis 快是一个系统工程结果，不是「内存」一个点决定的。

    3. 常用数据结构与实战映射¶

    3.1 String¶

    • 用途：缓存对象、计数器、分布式锁标记、简单状态位。
    • 常见命令：SET、GET、INCR、DECR、SETEX、SET key value NX EX 30。
    • 注意：对象序列化体积别太大，避免出现大 value。

    3.2 Hash¶

    • 用途：对象字段存储（如用户 profile、配置项）。
    • 优势：比一个对象一个 key 更易做字段级更新。
    • 命令：HSET、HGET、HMGET、HINCRBY。

    3.3 List¶

    • 用途：简单队列、时间线。
    • 命令：LPUSH、RPUSH、LPOP、RPOP、BLPOP。
    • 注意：积压严重时，消费延迟会上升；更复杂 MQ 建议用 Kafka/RabbitMQ。

    3.4 Set / ZSet¶

    • Set：去重集合（标签、共同关注）。
    • ZSet：带分数排序（排行榜、延迟队列）。
    • 命令：SADD、SISMEMBER、ZADD、ZRANGE、ZREVRANGE。

    3.5 Stream（进阶）¶

    • 用途：可消费组的消息流，支持 ACK 和重放。
    • 适合：中轻量异步任务链路；需要比 List 更强的消费管理能力。

    4. 缓存设计：命中率与一致性¶

    4.1 经典旁路缓存（Cache Aside）¶

    1. 读：先查 Redis，miss 再查 DB 并回填缓存。
    2. 写：先写 DB，再删缓存（常见策略）。

    4.2 常见问题与解法¶

    • 缓存穿透（查不存在数据）
    • 解法：缓存空值（短 TTL）、布隆过滤器、参数校验。
    • 缓存击穿（热点 key 过期瞬间大量并发打 DB）
    • 解法：热点不过期、互斥重建（单飞）、逻辑过期 + 异步刷新。
    • 缓存雪崩（大量 key 同时失效）
    • 解法：TTL 加随机抖动、分批过期、多级缓存、限流降级。

    4.3 一致性取舍¶

    • 缓存和 DB 很难做到严格实时一致，通常追求「可接受窗口内最终一致」。
    • 核心是：保证关键链路不会因为缓存失效导致 DB 被压垮。

    5. 过期与淘汰：内存不是无限的¶

    5.1 过期机制（TTL）¶

    • Redis 使用惰性删除 + 定期删除组合处理过期 key。
    • 不要依赖「到点必删」的强实时语义，只能保证「最终会删」。

    5.2 内存淘汰策略（maxmemory-policy）¶

    • 常见策略：
    • noeviction：不淘汰，写入报错。
    • allkeys-lru / allkeys-lfu：从所有 key 淘汰。
    • volatile-lru / volatile-lfu：仅从设置了 TTL 的 key 淘汰。
    • 推荐：缓存场景常用 allkeys-lru 或 allkeys-lfu（按业务验证）。

    6. 持久化：RDB 与 AOF¶

    6.1 RDB（快照）¶

    • 特点：某时刻数据快照，恢复速度通常快，文件紧凑。
    • 风险：两次快照之间的写入可能丢失。

    6.2 AOF（追加日志）¶

    • 特点：记录写命令，重放恢复，数据安全性通常更好。
    • appendfsync 策略：
    • always：最安全，性能开销大。
    • everysec：常用折中（可能丢 1 秒左右数据）。
    • no：由 OS 决定刷盘时机。

    6.3 组合建议¶

    • 线上常见：RDB + AOF 同开，平衡恢复速度和数据安全。

    7. 高可用与扩展¶

    7.1 主从复制¶

    • 主节点写，从节点异步复制；读扩展容易，但有复制延迟。
    • 业务上要区分「可读旧数据」和「必须读最新」场景。

    7.2 Sentinel（哨兵）¶

    • 提供故障检测与自动主从切换。
    • 注意：故障切换期间客户端可能出现短暂写失败或重连抖动。

    7.3 Cluster（分片）¶

    • 通过 16384 槽分片扩展容量与吞吐。
    • 多 key 操作受限（需同槽或使用 hash tag）。
    • 不是强一致分布式数据库，设计时要明确一致性边界。

    8. Java 开发落地（Spring Boot）¶

    • 客户端常用：Lettuce（Netty，线程安全，Spring Data Redis 默认常见选择）。
    • 序列化建议：明确 key/value 序列化格式（JSON 或高效二进制），避免 JDK 默认序列化导致可读性差和体积膨胀。
    • 连接池：避免池太大；Redis 单实例处理能力高，过多连接可能反而增加上下文切换与网络压力。
    • Pipeline：批量命令可显著降低 RTT 成本，但要控制一次批量大小，防止响应包过大。
    • 事务：MULTI/EXEC 不是关系型事务替代；跨 key 强一致仍应优先依赖数据库事务设计。

    9. 分布式锁：能用，但要谨慎¶

    最小可用写法（单实例）：

    • 加锁：SET lock_key requestId NX EX 10
    • 解锁：用 Lua 脚本保证「比较 requestId 后再删除」，避免误删别人的锁。

    注意点：

    • 锁超时要根据业务最长执行时间设置，并考虑续约机制。
    • Redis 锁适合「降低并发冲突」，不适合承载金融级绝对一致。
    • 复杂场景优先：数据库约束 + 幂等 + 补偿机制。

    10. 性能与稳定性排查 checklist¶

    • 慢命令：SLOWLOG GET，重点看大 key 操作、KEYS、大范围扫描。
    • 热点 key：某个 key QPS 过高会形成单点瓶颈，需分片或本地缓存分担。
    • 大 key：删除和迁移成本高，可能阻塞；优先拆分数据模型。
    • 内存碎片率：关注 INFO memory 的碎片指标，必要时重启/迁移整理。
    • 延迟监控：区分网络、客户端线程池、Redis 自身 CPU/内存瓶颈。
    • 命令禁用：生产避免误用 FLUSHALL、KEYS * 等高风险命令。

    11. 面试与实战高频题（速记）¶

    • Redis 为什么快：内存 + 单线程执行路径 + I/O 多路复用 + 高效数据结构。
    • 缓存三大问题：穿透、击穿、雪崩及对应治理手段。
    • 持久化怎么选：RDB、AOF、everysec 折中。
    • 如何做分布式锁：SET NX EX + Lua 原子释放 + 过期与续约设计。
    • 主从/哨兵/集群区别：高可用 vs 扩展能力 vs 一致性边界。

    12. 小结：Redis 在系统中的正确角色¶

    • Redis 解决的是「高并发下低延迟读写与协调问题」。
    • 真正稳定的系统依赖：缓存策略、数据库兜底、限流熔断、监控告警一起配合。
    • 技术决策核心不是「能不能用 Redis」，而是「故障与不一致发生时，业务能否优雅退化」。
