    Java基础面试题问卷¶

    第一题：请解释什么是面向对象，以及它解决了什么问题？¶

    30秒快速回答¶

    面向对象是一种编程范式，把程序要处理的事物看作"对象"，每个对象包含数据（属性）和行为（方法）。你可以这样理解：传统的面向过程像是给你一个做菜的菜谱（一步一步怎么做），而面向对象像是告诉你有一个"厨师"对象能做菜。

    它解决的问题： - 代码结构混乱：大型项目的面向过程代码逻辑复杂、难维护 - 代码重用困难：需要复制粘贴相似代码 - 修改影响范围大：改一个地方容易影响其他地方

    深入追问¶

    1. 面向对象的三大特性各是什么？能用日常生活的例子解释吗？

    标准答案： - 封装：把相关的数据和操作放在一起，对外只暴露需要的接口。比如：你用手机缴话费，不需要知道后台怎么扣费、怎么统计，只管输入号码和金额。 - 继承：子类继承父类的属性和方法，形成IS-A关系。比如：猫是动物，猫继承了动物的"会吃饭、会睡觉"等特性。 - 多态：同一个方法在不同对象上有不同表现。比如：不同动物都有"叫"这个方法，但狗叫"汪汪"，猫叫"喵喵"。

    1. 继承和多态要配合使用，为什么？

    标准答案： 多态需要两个前提：继承和方法重写。只有建立了继承关系，子类才能重写父类的方法，这样用父类引用指向不同的子类对象时，就能调用不同的方法实现。这样做的好处是：你写的代码不用知道具体是哪个子类，只需要调用父类的方法就行，新增子类时也不用改源代码。

    项目示例¶

    // 想象你在开发支付系统：

// 不用多态（代码坏）：
if (type.equals("支付宝")) {
    new 支付宝支付().pay();
} else if (type.equals("微信")) {
    new 微信支付().pay();
} else if (type.equals("银行卡")) {
    new 银行卡支付().pay();
}
// 新增一个支付方式就要改代码！

// 用多态（代码好）：
Payment payment = getPayment(type);  // 返回具体的支付方式
payment.pay();  // 直接调用，不关心具体是哪个支付方式
// 新增支付方式，只需新建类实现Payment接口，主代码无需改动！

    第二题：Java中基本数据类型和String有什么区别？为什么String要设计成不可变的？¶

    30秒快速回答¶

    基本数据类型（8种）直接存储值，如 int i = 10; 存的就是数字10。
    String 是引用类型，存的是对象的地址。

    String为什么不可变： String被声明为 final，内部的 char[] 数组也是 final，一旦创建就不能改变。不可变有四大好处（可以背这个）。

    90秒详细解答¶

    String不可变有什么好处？¶

    1. 能缓存hash值：HashMap用String做key时，hash值永远不变，只需算一次，性能好。
    2. String Pool的需要：相同内容的字符串可以复用同一个对象，节省内存。
    3. 安全性：String经常做参数，如果可变，参数在传递过程中被改了就危险了（比如网络连接地址被改）。
    4. 线程安全：天生具备，多个线程可以安全地使用同一个String，不用加锁。

    深入追问¶

    1. 那 String a = "abc"; String b = "abc"; 这两个引用指向同一个对象吗？

    标准答案： 是的，都指向String常量池里的同一个对象。因为 "abc" 这样用双引号写的字符串会自动放入常量池。所以 a == b 的结果是 true。

    但如果是 String a = new String("abc"); String b = new String("abc"); 就是两个不同的对象，a == b 是 false。

    重要提示： 实际代码中判断String相等要用 equals() 方法，不要用 ==！

    1. String + 拼接和StringBuilder有什么区别？

    标准答案： - String a = "hello" + "world"; 每个 + 都会产生一个新的String对象，效率低。 - StringBuilder 是可变的，多次append后只生成一个对象，效率高。 - StringBuffer 和StringBuilder一样，但线程安全（因为加了锁），所以慢一点。

    建议： 循环拼接或大量拼接用StringBuilder；只有在多线程环境才考虑StringBuffer。

    项目示例¶

    // 常见错误：
String result = "";
for (int i = 0; i < 10000; i++) {
    result = result + "数据";  // 这会创建10000个String对象！性能灾难
}

// 改进方案：
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 10000; i++) {
    sb.append("数据");  // 只创建一个对象，效率高100倍
}
String result = sb.toString();

    第三题：Java中函数参数传递是值传递还是引用传递？¶

    30秒快速回答¶

    答：始终是值传递。

    即使传递的是对象，本质上是将对象的地址以值的方式复制给形参。两个指针（实参和形参）在不同的内存位置存放相同的地址值。

    90秒详细解答¶

    // 场景1：传递基本类型
public void changeValue(int a) {
    a = 100;  // 只改变形参，不影响实参
}
int x = 10;
changeValue(x);  // x还是10

// 场景2：传递对象
public void changeObject(Dog dog) {
    dog.name = "旺财";  // 通过地址改对象内容，实参dog对象也改了！
    dog = new Dog("小黑");  // 但这行改变的只是形参指向，不影响实参
}
Dog dog = new Dog("大黄");
changeObject(dog);  // dog.name 现在是 "旺财"
                    // dog 仍然是原来的Dog对象，没有变成"小黑"

    关键点： - 改对象的属性值会影响实参（因为指向同一个对象） - 改形参的引用指向不会影响实参（因为是两个独立的指针）

    深入追问¶

    1. 如果想在函数内改变实参的引用怎么办？

    标准答案： Java无法做到。因为只有值传递，形参是实参的"复制品"，改形参的指向改不了实参的指向。

    解决办法：返回新对象，或用容器类（如List）包装。

    项目示例¶

    // 常见误解导致的bug：
public void initUser(User user) {
    if (user == null) {
        user = new User();  // 这行无法修改外面的user引用！
    }
    user.setName("张三");
}

User u = null;
initUser(u);  // u还是null，会导致NPE异常

// 改为：
public User initUser(User user) {
    if (user == null) {
        user = new User();
    }
    user.setName("张三");
    return user;  // 返回初始化后的对象
}

User u = null;
u = initUser(u);  // 这样才能让u指向新对象
