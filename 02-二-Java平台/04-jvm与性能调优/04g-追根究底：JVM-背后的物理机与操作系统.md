# 03 JVM与性能调优（零基础版） - 追根究底：JVM 背后的物理机与操作系统

## 本节核心约束（先读这个）
- 追根究底：JVM 背后的物理机与操作系统 的本质，是在 CPU、内存、磁盘、网络四类成本之间做可控取舍。

## 物理基础（建立直觉）
- 本地计算通常远快于跨线程同步，跨线程同步又远快于跨网络调用。
- 随机 IO 和阻塞等待往往比纯 CPU 计算更容易把接口拖慢。
- 一旦请求量放大，平时不明显的锁冲突、连接等待、缓存失效会被迅速放大。

## 六段因果链

### 1) 约束来源
- 追根究底：JVM 背后的物理机与操作系统 面对的不是“能不能做”，而是“在什么成本下做”。
- 没有成本视角时，结论很容易变成背题。

### 2) 问题场景
- 当业务从日常 QPS 300 增长到峰值 QPS 3000 时，追根究底：JVM 背后的物理机与操作系统 相关的瓶颈会开始显性化。
- 当数据量从十万级上升到千万级后，原先简单可用的方法常会失效。

### 3) 概念诞生
- 追根究底：JVM 背后的物理机与操作系统 相关机制的设计目标，是把高代价操作变少、变短、变可观测。
- 工程实现通常不是追求完美，而是追求“在当前约束下足够稳定”。

### 4) 解决与未解决
- 解决了：能把问题放到真实代价上讨论，而不是只背定义。
- 没解决：不能自动消灭所有瓶颈，仍要结合业务场景和数据分布判断。
- 新复杂度：需要引入监控、压测和边界条件说明，学习与维护成本会上升。

### 5) 场景选型（含失效条件）
| 场景 | 推荐方案 | 核心理由 | 失效条件 |
|------|----------|----------|----------|
| 日常低并发 | 先用简单方案 | 复杂度低、回报快 | 峰值流量突然放大 |
| 读多写少 | 优先减少回表和远程调用 | 延迟更稳定 | 写放大或数据失真 |
| 核心交易 | 优先保正确与可回滚 | 可控性更强 | 资源耗尽且无降级 |

### 6) 高并发变化
- 追根究底：JVM 背后的物理机与操作系统 在高并发下要先定义失效条件，再定义扩容和降级顺序。
- 容量估算：并发约等于 QPS 乘 RT。示例：QPS=4000、RT=0.12 秒，则并发约 480。

## 持续观测与线上排障

### 关键指标
| 指标 | 正常范围 | 告警阈值 | 采集方式 | 说明 |
|------|----------|----------|----------|------|
| P95延迟 | 小于 100ms | 大于 200ms | APM/网关指标 | 看体验是否劣化 |
| 错误率 | 小于 0.5% | 大于 1% | 应用日志/网关 | 看稳定性 |
| 资源等待时间 | 小于 10ms | 大于 50ms | 线程池/连接池指标 | 看瓶颈是否外溢 |

### 排查命令
- top -H -p <pid>：看热点线程。
- jstack <pid> | head -n 200：看阻塞与锁等待。
- vmstat 1 5：看系统整体调度和上下文切换。


## 追根究底：JVM 背后的物理机与操作系统

### 1) 为什么对象分配快，但 Full GC 很痛
### 是什么
- 大多数对象在堆上按指针碰撞快速分配，速度接近顺序写内存。

### 为什么
- 年轻代回收快是因为“多数对象很快死亡”；但老年代回收需要遍历和整理，成本高。

### 怎么做
- 控制对象生命周期，减少短时大量临时对象。
- 减少大对象直接进入老年代的概率。

### 高并发如何选择
- 低延迟业务优先降低分配抖动和晋升压力，而不是盲目增大堆。

### 面试题与标准答案
- 问：为什么堆调大了，有时反而更卡？
- 答：堆变大降低了回收频率，但单次回收扫描范围变大，停顿可能更长。

### 2) CPU 缓存与指令流水线如何影响 Java 性能
### 是什么
- CPU 真正执行时依赖缓存和流水线，不是逐条慢慢执行字节码。

### 为什么
- 分支预测失败、缓存未命中、内存随机访问都会让 CPU 空转等待。

### 怎么做
- 热路径数据结构尽量连续访问，减少随机指针跳转。
- 避免在核心循环里做复杂对象装箱拆箱。

### 高并发如何选择
- 对延迟敏感场景，优先优化缓存命中率和对象布局。

### 面试题与标准答案
- 问：为什么同样 O(n) 的两段代码，耗时差很多？
- 答：大 O 只描述增长趋势，真实性能还受缓存命中、分支预测、常数项影响。

### 3) 线程阻塞为什么会放大延迟
### 是什么
- 线程被阻塞时会触发内核调度，发生上下文切换。

### 为什么
- 切换有系统开销，阻塞多了会造成线程池排队和级联超时。

### 怎么做
- 区分 CPU 密集和 IO 密集线程池。
- 为外部依赖设置超时，避免线程长期占用。

### 高并发如何选择
- 热链路以“少阻塞、少等待”为先，必要时异步化。

### 面试题与标准答案
- 问：线上线程池满了最先看什么？
- 答：先看任务类型和阻塞点，是慢 SQL、慢 RPC 还是锁等待，再决定扩容、限流或降级。
