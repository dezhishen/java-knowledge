15 根因视角-物理机操作系统数据结构

15 根因视角：物理机 / 操作系统 / 数据结构

你为什么会感觉“雾里看花”

  • 因为很多资料只讲 API，不讲 API 背后的约束。
  • 高级面试真正考的是：这个方案为什么成立，在哪些边界下会失效。

一张总图：程序性能从哪来

  1. 物理机层：CPU、缓存、内存、磁盘、网卡决定硬上限。
  2. 操作系统层：线程调度、系统调用、网络模型决定运行成本。
  3. 数据结构层：组织方式决定时间复杂度和常数开销。
  4. 框架层：把上面三层能力封装成可用组件。

A. 物理机层（CPU/内存/磁盘）

1. CPU 缓存与缓存行

  • 是什么：CPU 不直接频繁访问主内存，而先访问 L1/L2/L3 缓存。
  • 为什么：缓存比内存快很多，命中率决定大量程序性能。
  • 怎么做：
    • 热数据尽量连续存放，减少缓存未命中。
    • 避免伪共享（不同线程频繁写同一缓存行）。
  • 高并发如何选择：热点计数器可分片，最后再汇总。

2. 指令重排与内存屏障

  • 是什么：CPU/编译器会在不改变单线程结果前提下调整执行顺序。
  • 为什么：为了提高流水线效率，但会影响多线程可见性。
  • 怎么做：
    • 关键共享变量用 volatile 或锁。
    • 理解 happens-before 约束。
  • 高并发如何选择：简单状态用 volatile，复合状态用锁。

3. 磁盘与顺序写

  • 是什么：顺序写远快于随机写。
  • 为什么：磁盘/SSD 的底层写入特性决定了 IO 代价。
  • 怎么做：
    • 日志型结构优先顺序追加。
    • 批量写减少刷盘次数。
  • 高并发如何选择：吞吐场景优先顺序写 + 批处理。

B. 操作系统层（调度/系统调用/网络）

1. 线程调度与上下文切换

  • 是什么：CPU 在多个线程之间切换执行。
  • 为什么：切换要保存/恢复寄存器和调度状态，成本不低。
  • 怎么做：
    • 线程数不要盲目增大。
    • 分离 CPU 密集与 IO 密集线程池。
  • 高并发如何选择：优先减少无效切换，而不是盲目加线程。

2. 锁与 futex

  • 是什么：Java 锁在竞争激烈时会落到内核等待/唤醒机制。
  • 为什么：内核参与意味着系统调用成本，竞争高就会变慢。
  • 怎么做：
    • 缩短临界区。
    • 降低共享状态。
  • 高并发如何选择：低冲突 CAS，高冲突分片锁或队列化。

3. epoll 与网络事件驱动

  • 是什么：单线程可监听大量连接的就绪事件。
  • 为什么：相比每连接一线程，事件驱动更省线程和内存。
  • 怎么做：
    • 用 Reactor/Netty 模型处理高并发连接。
    • 业务处理和 IO 线程分离。
  • 高并发如何选择：连接多、IO 重场景优先事件驱动模型。

C. 数据结构层（复杂度与实际开销）

1. 哈希表

  • 是什么：平均 O(1) 查找。
  • 为什么：哈希冲突和扩容会带来抖动。
  • 怎么做：
    • 设计均匀 key。
    • 控制负载因子。
  • 高并发如何选择：并发读写优先 ConcurrentHashMap。

2. B+Tree

  • 是什么：数据库索引常用结构，适合范围查询。
  • 为什么：和磁盘页组织匹配，减少 IO 次数。
  • 怎么做：
    • 索引列按查询模式设计。
    • 优先覆盖索引。
  • 高并发如何选择：热点写场景避免单索引热点。

3. 队列

  • 是什么：先进先出，天然适合削峰。
  • 为什么：把峰值请求转成平滑消费。
  • 怎么做：
    • 有界队列 + 拒绝策略。
    • 监控堆积深度与消费延迟。
  • 高并发如何选择：核心链路队列长度必须可控，防止延迟雪崩。

面试高频追问与标准答案

  • 问：为什么并发问题常常“本地没事，线上出事”？
  • 答：本地并发度低且负载简单，线上在 CPU 缓存一致性、线程调度竞争、IO 抖动叠加下，隐藏问题会放大。
  • 问：为什么线程不是越多越快？
  • 答：线程过多会带来调度和切换成本，还会争抢 CPU 缓存和锁，导致整体吞吐下降。
  • 问：为什么数据库索引常用 B+Tree 而不是红黑树？
  • 答：B+Tree 更适合磁盘页访问和范围扫描，能显著减少 IO 次数。

学习落地法（每天45分钟）

  1. 15分钟：读一个底层概念并画因果链。
  2. 15分钟：用一个线上案例解释该概念。
  3. 15分钟：回答一个面试追问并录音复盘。
