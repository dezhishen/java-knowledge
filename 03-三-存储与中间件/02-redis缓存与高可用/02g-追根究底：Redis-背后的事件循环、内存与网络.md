# 06 Redis 缓存与高可用（零基础版） - 追根究底：Redis 背后的事件循环、内存与网络

## 本节核心约束（先读这个）
- 高可用要解决的是单点故障不可避免，系统必须在局部失效时继续提供核心能力。

## 物理基础（建立直觉）
- 本地计算通常远快于跨线程同步，跨线程同步又远快于跨网络调用。
- 随机 IO 和阻塞等待往往比纯 CPU 计算更容易把接口拖慢。
- 一旦请求量放大，平时不明显的锁冲突、连接等待、缓存失效会被迅速放大。

## 六段因果链

### 1) 约束来源
- 追根究底：Redis 背后的事件循环、内存与网络 面对的不是“能不能做”，而是“在什么成本下做”。
- 没有成本视角时，结论很容易变成背题。

### 2) 问题场景
- 当业务从日常 QPS 300 增长到峰值 QPS 3000 时，追根究底：Redis 背后的事件循环、内存与网络 相关的瓶颈会开始显性化。
- 当数据量从十万级上升到千万级后，原先简单可用的方法常会失效。

### 3) 概念诞生
- 追根究底：Redis 背后的事件循环、内存与网络 相关机制的设计目标，是把高代价操作变少、变短、变可观测。
- 工程实现通常不是追求完美，而是追求“在当前约束下足够稳定”。

### 4) 解决与未解决
- 解决了：能把问题放到真实代价上讨论，而不是只背定义。
- 没解决：不能自动消灭所有瓶颈，仍要结合业务场景和数据分布判断。
- 新复杂度：需要引入监控、压测和边界条件说明，学习与维护成本会上升。

### 5) 场景选型（含失效条件）
| 场景 | 推荐方案 | 核心理由 | 失效条件 |
|------|----------|----------|----------|
| 日常低并发 | 先用简单方案 | 复杂度低、回报快 | 峰值流量突然放大 |
| 读多写少 | 优先减少回表和远程调用 | 延迟更稳定 | 写放大或数据失真 |
| 核心交易 | 优先保正确与可回滚 | 可控性更强 | 资源耗尽且无降级 |

### 6) 高并发变化
- 追根究底：Redis 背后的事件循环、内存与网络 在高并发下要先定义失效条件，再定义扩容和降级顺序。
- 容量估算：并发约等于 QPS 乘 RT。示例：QPS=4000、RT=0.12 秒，则并发约 480。

## 持续观测与线上排障

### 关键指标
| 指标 | 正常范围 | 告警阈值 | 采集方式 | 说明 |
|------|----------|----------|----------|------|
| P95延迟 | 小于 100ms | 大于 200ms | APM/网关指标 | 看体验是否劣化 |
| 错误率 | 小于 0.5% | 大于 1% | 应用日志/网关 | 看稳定性 |

- top -H -p <pid>：看热点线程。
- jstack <pid> | head -n 200：看阻塞与锁等待。
- vmstat 1 5：看系统整体调度和上下文切换。


## 追根究底：Redis 背后的事件循环、内存与网络

### 1) Redis 为什么快，但也会被单点热点拖慢
### 是什么
- Redis 核心命令执行模型偏单线程事件循环，避免了大量锁竞争。

### 为什么
- 单线程让执行路径短、开销小；但单 key 热点会独占处理时间。

### 怎么做
- 热点 key 分片（如 user:123 拆成多个桶）。
- 大 key 拆分，避免一次操作阻塞太久。

### 高并发如何选择
- 高热点场景优先“拆热点 + 本地缓存兜底”。

### 面试题与标准答案
- 问：Redis 单线程是不是性能瓶颈？
- 答：大多数场景不是，瓶颈常在网络、内存带宽或热点 key；但极端热点会成为单线程处理瓶颈。

### 2) 为什么内存淘汰会引发抖动
### 是什么
- 内存接近上限时，Redis 会按淘汰策略删除键值。

### 为什么
- 淘汰和回源叠加会导致命中率波动，数据库压力瞬时上升。

### 怎么做
- 保留足够内存余量。
- 热点数据设置更稳定过期策略并提前预热。

### 高并发如何选择
- 对核心业务优先保证命中稳定性，而不是单纯压榨内存利用率。

### 面试题与标准答案
- 问：为什么明明 Redis 在，数据库还是被打满？
- 答：常见是热点失效、淘汰抖动或穿透流量导致大规模回源。

### 3) 网络 I/O 和序列化为什么影响缓存收益
### 是什么
- 缓存访问不仅是 Redis 命令，还包含网络传输与序列化反序列化。

### 为什么
- 大对象或低效序列化会放大 CPU 与网络开销，抵消缓存速度优势。

### 怎么做
- 控制 value 大小，避免超大对象。
- 选更高效的序列化方案，必要时做字段裁剪。

### 高并发如何选择
- 低延迟链路优先短小数据结构与批量管道，减少往返次数。

### 面试题与标准答案
- 问：为什么加了 Redis 后 RT 仍不理想？
- 答：可能瓶颈转移到了网络和序列化成本，而不是 Redis 命令执行本身。
