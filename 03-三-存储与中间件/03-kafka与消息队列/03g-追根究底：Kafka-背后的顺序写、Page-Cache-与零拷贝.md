## 追根究底：Kafka 背后的顺序写、Page Cache 与零拷贝

### 1) Kafka 为什么吞吐高
### 是什么
- Kafka 以分区日志为核心，消息主要是顺序追加写。

### 为什么
- 顺序写对存储介质和操作系统缓存更友好，单位时间能写入更多数据。

### 怎么做
- 合理设置批量发送和压缩。
- 按业务键分区，避免极端热点分区。

### 高并发如何选择
- 吞吐优先时增加分区和批量；顺序优先时固定分区并控制并发。

### 面试题与标准答案
- 问：Kafka 为什么通常比传统队列吞吐更高？
- 答：核心在顺序写日志 + 批量传输 + 操作系统缓存协同，减少随机 IO 和系统调用。

### 2) Page Cache 与刷盘的取舍
### 是什么
- 写入先进入操作系统页缓存，再按策略刷到磁盘。

### 为什么
- 先写缓存可提高吞吐，但异常断电时存在少量未落盘风险。

### 怎么做
- 根据业务可靠性要求设置 ACK、副本数、最小同步副本。
- 核心 topic 提高可靠性参数，普通日志 topic 更偏吞吐。

### 高并发如何选择
- 资金类或订单类消息优先可靠性；行为埋点优先吞吐和成本。

### 面试题与标准答案
- 问：为什么提高可靠性参数会降低吞吐？
- 答：因为需要等待更多副本确认，网络与磁盘同步成本上升。

### 3) 零拷贝为什么能降 CPU
### 是什么
- 数据在内核态尽量直接从磁盘缓存到网卡，减少用户态拷贝。

### 为什么
- 少一次数据复制和上下文切换，就少一部分 CPU 开销。

### 怎么做
- 使用 Kafka 默认高效传输路径，避免业务层无谓二次拷贝。
- 控制消息体大小，避免大包导致网络抖动。

### 高并发如何选择
- 大流量分发场景优先批量+压缩+合理消息大小。

### 面试题与标准答案
- 问：零拷贝是“完全不拷贝”吗？
- 答：不是，它是减少不必要的数据拷贝与上下文切换，重点是降低 CPU 消耗。
