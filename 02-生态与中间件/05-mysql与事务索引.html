05 MySQL 与事务索引（零基础版）

先用人话说清楚

  • 数据库像图书馆，索引像目录。
  • 有目录找书很快，没目录就得从头翻到尾。

术语翻译

  • 索引：加速查找的数据结构。
  • 回表：先通过二级目录找到编号，再回主目录拿完整内容。
  • 事务：一组操作一起成功或失败。
  • MVCC：让读和写少打架的版本机制。
  • 死锁：你等我、我等你，谁都不动。

1. 索引

是什么

  • 用 B+Tree 组织键值，提升查询效率。

为什么

  • 没有索引时，全表扫描会拖垮数据库。

怎么做

  • 常用查询条件要建索引。
  • 用 EXPLAIN 看是否命中索引。

高并发如何选择

  • 优先覆盖索引，减少回表和随机 IO。

面试题与标准答案

  • 问：为什么加了索引还慢？
  • 答：可能没命中、回表太多、排序分组没走索引或返回数据过大。

2. 事务与隔离

是什么

  • 控制并发读写时的数据正确性。

为什么

  • 并发写入下，不加控制会出现脏读、不可重复读等问题。

怎么做

  • 事务尽量短。
  • 更新顺序统一，减少锁冲突。

高并发如何选择

  • 写冲突高时考虑分片或队列化写入。

面试题与标准答案

  • 问：为什么长事务危险？
  • 答：会长时间持锁，阻塞其他请求，放大延迟和超时风险。

3. 死锁与慢 SQL

是什么

  • 死锁是并发锁冲突；慢 SQL 是执行代价过高。

为什么

  • 二者都直接影响核心接口可用性。

怎么做

  • 固化死锁排查流程。
  • 建慢 SQL 日志和定期治理。

高并发如何选择

  • 热点写场景优先拆热点，避免集中锁竞争。

面试题与标准答案

  • 问：遇到死锁先做什么？
  • 答：先重试失败事务并收集死锁日志，定位冲突 SQL 和访问顺序。

本章30秒复述

  • 索引解决查找速度。
  • 事务解决并发正确性。
  • 高并发下要重点治理锁冲突和慢 SQL。

追根究底：MySQL 背后的页结构与操作系统 IO

1) 为什么 B+Tree 适合数据库而不是红黑树

是什么

  • B+Tree 按页组织，一个节点能放很多键，树高更低。

为什么

  • 磁盘/SSD 访问成本高，树高越低，随机 IO 次数越少。

怎么做

  • 索引列按查询条件和排序条件设计。
  • 尽量走覆盖索引，减少回表。

高并发如何选择

  • 读多场景优先覆盖索引，写多场景控制索引数量避免写放大。

面试题与标准答案

  • 问：为什么索引太多会拖慢写入？
  • 答：每次写入都要维护多个索引结构，触发更多页分裂和日志写入。

2) 为什么随机写会拖垮吞吐

是什么

  • 随机写会导致更多页定位和刷盘操作，顺序写更友好。

为什么

  • 存储设备对顺序访问更高效，随机访问会放大 IO 等待。

怎么做

  • 批量写、顺序主键、减少热点更新。
  • 合理设置刷盘策略，平衡安全和性能。

高并发如何选择

  • 峰值写入优先批处理和异步削峰，避免同步刷盘放大延迟。

面试题与标准答案

  • 问：为什么自增主键通常比随机 UUID 更友好？
  • 答：自增主键更接近顺序写，页分裂少，B+Tree 维护成本更低。

3) 锁等待为什么会级联扩散

是什么

  • 一条慢事务持锁时间过长，会阻塞后续事务，形成排队。

为什么

  • 事务和连接数是有限资源，被占满后上游请求会持续堆积。

怎么做

  • 缩短事务，避免事务内做外部调用。
  • 统一更新顺序，降低死锁概率。

高并发如何选择

  • 核心写链路优先短事务 + 幂等重试，必要时分片拆热点。

面试题与标准答案

  • 问：为什么“慢 SQL”会让整个系统抖动？
  • 答：慢 SQL 会占住连接和锁，导致线程池等待、超时重试、流量回放，最终形成连锁反应。
