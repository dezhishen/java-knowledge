06 Redis 缓存与高可用（零基础版）

先用人话说清楚

  • Redis 像超快记事本，把常用数据放在内存里，读写更快。
  • 但内存有限，策略不当会出大问题。

术语翻译

  • 缓存命中：要的数据在缓存里，直接拿到。
  • 穿透：请求的数据根本不存在，缓存挡不住。
  • 击穿：某个热点数据刚好过期，大量请求打到数据库。
  • 雪崩：很多缓存同一时间失效，数据库被瞬间打爆。

1. 缓存为什么快

是什么

  • Redis 用内存存数据，访问成本低。

为什么

  • 可以显著降低数据库压力和响应时间。

怎么做

  • 定义好 key 命名和过期时间。
  • 给热点数据做预热。

高并发如何选择

  • 热点数据用本地缓存 + Redis 双层结构。

面试题与标准答案

  • 问：为什么缓存能提速？
  • 答：因为内存访问远快于磁盘数据库，大量读请求可在缓存层直接消化。

2. 一致性与三大问题

是什么

  • 缓存和数据库可能不一致，并可能出现穿透、击穿、雪崩。

为什么

  • 缓存策略不合理会把压力反冲数据库。

怎么做

  • 空值缓存和布隆过滤器防穿透。
  • 互斥锁或逻辑过期防击穿。
  • 随机过期和分批重建防雪崩。

高并发如何选择

  • 先保数据库，再保缓存命中率。

面试题与标准答案

  • 问：缓存一致性常见做法？
  • 答：常用先更新数据库再删除缓存，并加重试保证删除成功率。

3. 高可用

是什么

  • 主从、哨兵、集群保障 Redis 故障后仍可服务。

为什么

  • 缓存层一旦不可用，主库压力会暴涨。

怎么做

  • 明确故障切换方案，定期演练。

高并发如何选择

  • 写压力大时优先分片扩容，避免单点瓶颈。

面试题与标准答案

  • 问：Redis 故障时你怎么止损？
  • 答：先限流和降级，保护数据库，再恢复缓存和热点数据。

本章30秒复述

  • Redis 负责快，数据库负责准。
  • 缓存问题本质是流量回源问题。
  • 高并发要先防雪崩，再谈命中率。

追根究底：Redis 背后的事件循环、内存与网络

1) Redis 为什么快，但也会被单点热点拖慢

是什么

  • Redis 核心命令执行模型偏单线程事件循环，避免了大量锁竞争。

为什么

  • 单线程让执行路径短、开销小；但单 key 热点会独占处理时间。

怎么做

  • 热点 key 分片（如 user:123 拆成多个桶）。
  • 大 key 拆分，避免一次操作阻塞太久。

高并发如何选择

  • 高热点场景优先“拆热点 + 本地缓存兜底”。

面试题与标准答案

  • 问：Redis 单线程是不是性能瓶颈？
  • 答：大多数场景不是，瓶颈常在网络、内存带宽或热点 key；但极端热点会成为单线程处理瓶颈。

2) 为什么内存淘汰会引发抖动

是什么

  • 内存接近上限时，Redis 会按淘汰策略删除键值。

为什么

  • 淘汰和回源叠加会导致命中率波动，数据库压力瞬时上升。

怎么做

  • 保留足够内存余量。
  • 热点数据设置更稳定过期策略并提前预热。

高并发如何选择

  • 对核心业务优先保证命中稳定性，而不是单纯压榨内存利用率。

面试题与标准答案

  • 问：为什么明明 Redis 在，数据库还是被打满？
  • 答：常见是热点失效、淘汰抖动或穿透流量导致大规模回源。

3) 网络 I/O 和序列化为什么影响缓存收益

是什么

  • 缓存访问不仅是 Redis 命令，还包含网络传输与序列化反序列化。

为什么

  • 大对象或低效序列化会放大 CPU 与网络开销，抵消缓存速度优势。

怎么做

  • 控制 value 大小，避免超大对象。
  • 选更高效的序列化方案，必要时做字段裁剪。

高并发如何选择

  • 低延迟链路优先短小数据结构与批量管道，减少往返次数。

面试题与标准答案

  • 问：为什么加了 Redis 后 RT 仍不理想？
  • 答：可能瓶颈转移到了网络和序列化成本，而不是 Redis 命令执行本身。
