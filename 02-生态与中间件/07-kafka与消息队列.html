07 Kafka与消息队列

07 Kafka 与消息队列（零基础版）

先用人话说清楚

  • 消息队列就像快递分拣中心：先收件排队，再按规则分发处理。
  • 它能把瞬时高峰“摊平”，避免后端一下被打爆。

术语翻译

  • Producer：发消息的人。
  • Consumer：收消息并处理的人。
  • Topic：消息主题分类。
  • Partition：一个主题下的分桶，用来并行处理。
  • Offset：消费进度编号。

1. MQ 的作用

是什么

  • 解耦、异步、削峰三大核心能力。

为什么

  • 没有 MQ，系统之间耦合太紧，峰值流量很容易压垮下游。

怎么做

  • 把非实时强依赖流程改为异步。
  • 给关键消息链路做重试和告警。

高并发如何选择

  • 峰值高时优先多分区提升吞吐。

面试题与标准答案

  • 问：为什么要引入 MQ？
  • 答：核心是解耦和削峰，把同步压力转成异步可控压力。

2. 重复和丢失

是什么

  • 消息系统可能重复、可能延迟，业务侧要可兜底。

为什么

  • 网络和机器都不可靠，必须按“会失败”来设计。

怎么做

  • 业务实现幂等：唯一键、去重表、状态机。
  • 失败进入死信队列，后续补偿处理。

高并发如何选择

  • 核心交易优先正确性，宁可慢一点也不乱账。

面试题与标准答案

  • 问：消息重复消费怎么办？
  • 答：业务侧做幂等，保证同一条消息处理多次结果一致。

3. 消费稳定性

是什么

  • 消费者扩缩容或故障会触发分区重分配。

为什么

  • 重平衡期间可能短暂停顿，延迟会变大。

怎么做

  • 控制单次拉取量和处理时长。
  • 监控消费延迟和堆积量。

高并发如何选择

  • 高吞吐优先批量消费，但要有超时和幂等保护。

面试题与标准答案

  • 问：如何避免消息堆积越来越严重？
  • 答：扩容消费者、优化处理逻辑、提高并行度，并对失败消息做隔离。

本章30秒复述

  • MQ 是解耦和削峰核心组件。
  • 消息可靠性靠业务幂等和补偿闭环。
  • 高并发场景要重点监控堆积和延迟。

追根究底：Kafka 背后的顺序写、Page Cache 与零拷贝

1) Kafka 为什么吞吐高

是什么

  • Kafka 以分区日志为核心，消息主要是顺序追加写。

为什么

  • 顺序写对存储介质和操作系统缓存更友好，单位时间能写入更多数据。

怎么做

  • 合理设置批量发送和压缩。
  • 按业务键分区，避免极端热点分区。

高并发如何选择

  • 吞吐优先时增加分区和批量；顺序优先时固定分区并控制并发。

面试题与标准答案

  • 问：Kafka 为什么通常比传统队列吞吐更高？
  • 答：核心在顺序写日志 + 批量传输 + 操作系统缓存协同，减少随机 IO 和系统调用。

2) Page Cache 与刷盘的取舍

是什么

  • 写入先进入操作系统页缓存，再按策略刷到磁盘。

为什么

  • 先写缓存可提高吞吐，但异常断电时存在少量未落盘风险。

怎么做

  • 根据业务可靠性要求设置 ACK、副本数、最小同步副本。
  • 核心 topic 提高可靠性参数，普通日志 topic 更偏吞吐。

高并发如何选择

  • 资金类或订单类消息优先可靠性；行为埋点优先吞吐和成本。

面试题与标准答案

  • 问：为什么提高可靠性参数会降低吞吐？
  • 答：因为需要等待更多副本确认，网络与磁盘同步成本上升。

3) 零拷贝为什么能降 CPU

是什么

  • 数据在内核态尽量直接从磁盘缓存到网卡，减少用户态拷贝。

为什么

  • 少一次数据复制和上下文切换，就少一部分 CPU 开销。

怎么做

  • 使用 Kafka 默认高效传输路径，避免业务层无谓二次拷贝。
  • 控制消息体大小，避免大包导致网络抖动。

高并发如何选择

  • 大流量分发场景优先批量+压缩+合理消息大小。

面试题与标准答案

  • 问：零拷贝是“完全不拷贝”吗？
  • 答：不是，它是减少不必要的数据拷贝与上下文切换，重点是降低 CPU 消耗。
